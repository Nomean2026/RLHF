DPO 训练代码讲解

当 AI 学会「自己当评委」— 从零拆解 Direct Preference Optimization

DPO（Direct Preference Optimization），是 2023 年斯坦福大学提出的一种对齐训练方法。

一句话总结：扔掉奖励模型，扔掉 Critic，只靠一组「好答案 vs 坏答案」的偏好数据，就能让模型学会人类的审美。

🧠 核心洞察：PPO 要训练一个奖励模型来当评委，再用 Critic 来估计基线——像请了一个评委团和一个记分员。DPO 发现：其实你可以直接用偏好数据来反推最优策略，根本不需要评委。 这就好比你不需要打分系统，只需要知道「这个比那个好」就够了。

🏗️ 整体架构：极简双模型

Policy 模型 πθ
⟷ 同一份偏好数据 ⟷
Reference 模型 πref 🔒 冻结


PPO 需要：Actor + Critic + Reward Model + Reference = 4 个模型
DPO 只需要：Policy + Reference = 2 个模型（Ref 还冻着！）

DPO 的天才之处在于：它证明了偏好学习可以直接在策略模型上完成，无需中间的奖励模型。

数学上，DPO 把 RLHF 的奖励函数 $r(y_w, y_l)$ 重参数化为策略和参考模型的对数概率比：

$$r(y_w, y_l) = \beta \log \frac{\pi_\theta(y_w)}{\pi_{\text{ref}}(y_w)} - \beta \log \frac{\pi_\theta(y_l)}{\pi_{\text{ref}}(y_l)}$$

翻译成人话：好答案相对于参考模型的「提升幅度」，减去坏答案的「提升幅度」，这个差值就是模型的「偏好分数」。

STEP 1 加载偏好数据

def load_dpo_data(filepath):
    """加载 DPO 偏好数据"""
    data = []
    with open(filepath, 'r', encoding='utf-8') as f:
        for line in f:
            sample = json.loads(line)
            chosen = sample["chosen"]
            rejected = sample["rejected"]
            # 转换为文本
            chosen_text = "".join([
                f"<|{t['role']}|>{t['content']}"
                for t in chosen
            ])
            rejected_text = "".join([
                f"<|{t['role']}|>{t['content']}"
                for t in rejected
            ])
            data.append((chosen_text, rejected_text))
    return data

STEP 1 加载偏好数据 — 建立「审美标准」

DPO 的训练数据长得像这样（JSONL 格式）：

{
  "chosen":   [{"role":"user","content":"写首诗"}, {"role":"assistant","content":"春风又绿江南岸..."}],
  "rejected": [{"role":"user","content":"写首诗"}, {"role":"assistant","content":"好的，这是诗：诗。"}]
}

每条数据包含一对：chosen（人类更喜欢的答案）和 rejected（人类不喜欢的答案）。

这就像你去餐厅吃饭，朋友指着两道菜说「这个好吃，那个难吃」。你不需要知道具体几分，只需要知道哪个更好。DPO 的数据就是这样——一组二选一的偏好对。

数据处理细节

原始数据是对话格式（多轮 message），代码把它们拼成一个字符串：

<|user|>写首诗<|assistant|>春风又绿江南岸...

这种 <|role|> 标记方式是很多对话模型的标准格式，用来告诉模型「这是谁在说话」。

⚠️ 注意：实际工程中，chosen 和 rejected 通常共享同一个 prompt（用户输入），只有 assistant 的回答不同。这让模型能精准对比「同一个问题下，哪种回答更好」。

STEP 2 初始化两个模型

def dpo_train():
    batch_size = 4
    epochs = 3
    lr = 1e-6
    beta = 0.1  # DPO 温度系数
    device = 'cuda' if torch.cuda.is_available() else 'cpu'
    max_len = 256

    tokenizer = Tokenizer.from_file("tokenizer_luxun/tokenizer.json")
    vocab_size = len(tokenizer.get_vocab())
    pad_id = tokenizer.token_to_id("<|pad|>")

    # 加载 SFT 模型作为 Policy，并复制 Reference
    policy = FullTransformer(
        vocab_size, d_model=64,
        n_layers=3, n_heads=4
    ).to(device)
    policy.load_state_dict(torch.load(
        "checkpoints_sft/sft_model.pth",
        map_location=device
    ))

    ref_model = FullTransformer(
        vocab_size, d_model=64,
        n_layers=3, n_heads=4
    ).to(device)
    ref_model.load_state_dict(torch.load(
        "checkpoints_sft/reference_model.pth",
        map_location=device
    ))

    # 🔒 冻结 Reference 模型！
    for param in ref_model.parameters():
        param.requires_grad = False

    optimizer = torch.optim.AdamW(
        policy.parameters(), lr=lr
    )

STEP 2 初始化两个模型 — 从 SFT 出发

两个模型都从 SFT（Supervised Fine-Tuning）检查点加载，这意味着它们的起点是同一个「会说人话」的模型。

Policy 模型：要训练的主角，会不断进化。

Reference 模型：冻结的「参照物」，永远不更新。

Reference 模型就像一面「不变的镜子」。Policy 模型每次调整后，都要照镜子看看自己「偏离了原来多远」。这防止模型为了迎合偏好而变得面目全非（模式崩塌）。

关键参数：beta = 0.1

$$\mathcal{L}_{\text{DPO}} = -\log\sigma\left(\beta \left[\log\frac{\pi_\theta(y_w)}{\pi_{\text{ref}}(y_w)} - \log\frac{\pi_\theta(y_l)}{\pi_{\text{ref}}(y_l)}\right]\right)$$

$\beta$ 控制「偏好强度」：

  • $\beta$ 越大 → 模型越「听话」，拼命拉开 chosen 和 rejected 的差距
  • $\beta$ 越小 → 模型越「佛系」，不太在意偏好信号
  • $\beta = 0.1$ 是论文推荐值，平衡了对齐效果和生成质量

⚠️ 为什么 Reference 也从 SFT 加载？

因为 DPO 的「参照物」应该是 SFT 之后的状态，而不是随机初始化的模型。这保证了对比的公平性——我们在 SFT 的基础上，进一步学习偏好。

STEP 3 计算对数概率

    def get_log_prob(model, text):
        """计算给定文本的对数概率（总和）"""
        ids = tokenizer.encode(text).ids
        if len(ids) > max_len:
            ids = ids[:max_len]
        ids_tensor = torch.tensor(
            [ids], device=device
        )
        logits = model(ids_tensor)  # [1, S, V]

        # 使用 cross_entropy 计算 token 级 log_prob
        shift_logits = logits[:, :-1, :]::contiguous()
        shift_labels = ids_tensor[:, 1:]::contiguous()
        loss = F.cross_entropy(
            shift_logits.view(-1, vocab_size),
            shift_labels.view(-1),
            reduction='none'
        )
        # 转换为 log_prob (sum)
        log_prob = -loss.sum()
        return log_prob

STEP 3 计算对数概率 — 给答案「打分」

这个函数是整个 DPO 的核心工具：给一段文本，返回模型认为它「有多合理」的对数概率总和。

想象你在做填空题。模型预测下一个词的概率分布，就像一个学生在「猜下一个字是什么」。如果整段话模型都能轻松预测（概率高），说明这段话在模型看来「很自然、很合理」。对数概率的总和，就是模型对这段话的「整体自信度」。

为什么要 shift？

shift_logits = logits[:, :-1, :] 和 shift_labels = ids_tensor[:, 1:]

这是语言模型的标准操作：用位置 $t$ 的预测去匹配位置 $t+1$ 的真实 token。

$$P(y_t \mid y_1, y_2, \ldots, y_{t-1})$$

第一个 token 没有「上文」可以预测它，所以我们错开一位。

为什么用 reduction='none' 再手动 .sum()？

因为 DPO 需要的是整段文本的对数概率总和，而不是平均值。这样可以保证不同长度的文本之间可以公平比较。

输出含义

返回的 log_prob 是一个负数（对数概率 ≤ 0）：

  • 越接近 0 → 模型觉得这段话越「理所当然」
  • 越小（越负） → 模型觉得这段话越「出乎意料」

STEP 4 计算四组对数概率

    policy.train()
    for epoch in range(epochs):
        total_loss = 0
        for chosen_text, rejected_text in train_data:
            # Policy 模型的对数概率
            logp_chosen_policy = get_log_prob(
                policy, chosen_text
            )
            logp_rejected_policy = get_log_prob(
                policy, rejected_text
            )
            # Reference 模型的对数概率（不需要梯度）
            with torch.no_grad():
                logp_chosen_ref = get_log_prob(
                    ref_model, chosen_text
                )
                logp_rejected_ref = get_log_prob(
                    ref_model, rejected_text
                )

STEP 4 计算四组对数概率 — 收集「证据」

这里一共算了 4 个值，形成一个 2×2 矩阵：

chosen（好答案）rejected（坏答案）
Policy 模型logp_chosen_policylogp_rejected_policy
Reference 模型logp_chosen_reflogp_rejected_ref
这就像让两个人（Policy 和 Reference）分别给两道菜（chosen 和 rejected）打分。通过对比他们的打分差异，我们就能知道 Policy 相对于 Reference「进步了多少」。

为什么要 torch.no_grad()？

Reference 模型是冻结的，不需要计算梯度。用 no_grad() 可以：

  • 节省显存（不需要存储反向传播的中间值）
  • 加速计算（少做很多无用功）
  • 保证 Reference 不会被意外更新

STEP 5 DPO 损失函数

            # DPO 损失
            diff_chosen = logp_chosen_policy - logp_chosen_ref
            diff_rejected = logp_rejected_policy - logp_rejected_ref

            loss = -F.logsigmoid(
                beta * (diff_chosen - diff_rejected)
            )

STEP 5 DPO 损失函数 — 最核心的一步

这是 DPO 的精华所在，只有两行代码，却浓缩了整篇论文的数学推导。

拆解公式

$$\text{diff\_chosen} = \log \frac{\pi_\theta(y_w)}{\pi_{\text{ref}}(y_w)}$$

这表示：Policy 相对于 Reference，对好答案「提升了多少」。如果 Policy 更喜欢好答案，这个值是正数。

$$\text{diff\_rejected} = \log \frac{\pi_\theta(y_l)}{\pi_{\text{ref}}(y_l)}$$

同理：Policy 对坏答案「提升了多少」。

$$\mathcal{L} = -\log\sigma\left(\beta \cdot (\text{diff\_chosen} - \text{diff\_rejected})\right)$$

最后：好答案的提升 减去 坏答案的提升，差值越大，loss 越小。

假设你是一个美食评论家（Policy），需要对比两道菜。你不能直接说「这道 9 分那道 3 分」，但你可以说「相对于我上次吃的（Reference），这道进步了 5 分，那道退步了 2 分」。差距越大，说明你的「品味」越明确。DPO 损失就是在最大化这个差距。

为什么用 logsigmoid 而不是 sigmoid？

logsigmoid(x) = log(sigmoid(x))

数学上等价，但数值更稳定。sigmoid(x) 在 $x$ 很大时会溢出到 1，取 log 后变成 0，梯度消失。直接用 logsigmoid 可以避免这个问题。

STEP 6 训练循环 — 稳步前进

训练循环非常标准：梯度清零 → 反向传播 → 参数更新。没有 PPO 那种复杂的「收集经验 → 计算 GAE → 更新 Critic → 更新 Actor」的流程。

DPO 的训练为什么这么简单？

  • 没有 Reward Model：不需要训练额外的模型来打分
  • 没有 Critic：不需要估计基线（baseline）
  • 没有 GAE：不需要计算广义优势估计
  • 没有采样：直接用偏好数据的 log 概率
  • 标准交叉熵：用 PyTorch 原生的 F.cross_entropy

整个训练和普通的监督学习几乎一样简单！

⚠️ 学习率 lr = 1e-6

为什么这么小？因为 DPO 是在 SFT 模型的基础上微调，太大的学习率会破坏已有的能力（灾难性遗忘）。

🏁 总结：DPO vs PPO

维度PPO（RLHF）DPO
模型数量4 个（Actor + Critic + Reward + Reference）2 个（Policy + Reference）
是否需要奖励模型是（先训练，再用）否
训练复杂度高（多阶段、多循环）低（单阶段、标准反向传播）
显存占用高（4 个模型 + 采样 buffer）低（2 个模型 + 梯度）
超参数多且敏感（clip ratio, GAE λ, ...）少（主要是 β）
理论基础策略梯度 + 奖励建模Bradley-Terry 偏好模型
效果强大但不稳定接近 PPO，更稳定

DPO 的哲学

DPO 告诉我们一个深刻的道理：人类的偏好不需要被「量化成分数」，只需要「成对比较」就够了。

就像你不需要给每道菜打 1-10 分，只需要说「这个比那个好吃」——DPO 用最少的信息，达到了几乎一样好的对齐效果。

🏆 一句话总结：PPO 像是请了一个美食评委团，DPO 像是直接问朋友「你更喜欢吃哪个」。效果差不多，但后者简单太多了。
