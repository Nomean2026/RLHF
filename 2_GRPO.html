GRPO：组内比武，谁行谁上

没有 Critic 的强化学习 —— 用同组回答的相对排名代替绝对价值估计

📄 grpo_luxun.py

# grpo_luxun.py - GRPO 训练
# 组内相对策略优化：对同一 prompt 采样多个回答，组内标准化奖励，无需 Critic。

import torch
import torch.nn.functional as F
import numpy as np
import os
from tokenizers import Tokenizer

def grpo_train():
    batch_size = 1    # 每次处理一个 prompt
    G = 4             # 每组回答数量（比武的参赛人数）
    epochs = 3
    lr = 1e-6
    clip_eps = 0.2   # PPO 裁剪阈值
    kl_coef = 0.05   # KL 散度系数（防止跑太远）
    device = 'cuda' if torch.cuda.is_available() else 'cpu'
    max_len = 256

    tokenizer = Tokenizer.from_file("tokenizer_luxun/tokenizer.json")
    vocab_size = len(tokenizer.get_vocab())
    eos_id = tokenizer.token_to_id("<|eos|>")
    # 加载 SFT 模型作为 Policy，Reference 冻结
    policy = FullTransformer(vocab_size, d_model=64, n_layers=3, n_heads=4).to(device)
    policy.load_state_dict(torch.load("checkpoints_sft/sft_model.pth", map_location=device))

    # Reference 模型：冻结的"原版"，用来算 KL 散度
    ref_model = FullTransformer(vocab_size, d_model=64, n_layers=3, n_heads=4).to(device)
    ref_model.load_state_dict(torch.load("checkpoints_sft/reference_model.pth", map_location=device))
    for param in ref_model.parameters():
        param.requires_grad = False

    optimizer = torch.optim.AdamW(policy.parameters(), lr=lr)

    # 奖励模型（轻量级，只打分）
    reward_model = RewardModel(FullTransformer(vocab_size, d_model=64, n_layers=3, n_heads=4)).to(device)
    prompts = ["<|user|>你好<|assistant|>", "<|user|>今天天气不错<|assistant|>"]

    policy.train()
    for epoch in range(epochs):
        for prompt in prompts:
            # ===== 第一步：采样 G 个回答 =====
            responses = []
            all_log_probs = []  # 旧策略下的 log_probs
            for _ in range(G):
                input_ids = tokenizer.encode(prompt).ids
                input_tensor = torch.tensor([input_ids], device=device)
                generated = []
                log_probs = []
                for step in range(50):
                    logits = policy(input_tensor)[:, -1, :]
                    probs = F.softmax(logits, dim=-1)
                    next_token = torch.multinomial(probs, num_samples=1)
                    generated.append(next_token.item())
                    log_prob = torch.log(probs[0, next_token.item()])
                    log_probs.append(log_prob)
                    input_tensor = torch.cat(
                        [input_tensor, next_token.unsqueeze(0)], dim=1
                    )
                    if next_token.item() == eos_id:
                        break
                responses.append(generated)
                all_log_probs.append(log_probs)
            # ===== 第二步：奖励模型打分 =====
            rewards = []
            for gen in responses:
                full_ids = torch.tensor(
                    [tokenizer.encode(prompt).ids + gen], device=device
                )
                r = reward_model(full_ids).item()
                rewards.append(r)
            # ===== 第三步：组内标准化得优势（代替 Critic） =====
            rewards_t = torch.tensor(rewards, device=device)
            mean_r = rewards_t.mean()
            std_r = rewards_t.std() + 1e-8
            advantages = (rewards_t - mean_r) / std_r
            # ===== 第四步：计算 GRPO 损失 =====
            total_loss = 0
            total_tokens = 0
            for i in range(G):
                advantage = advantages[i]
                if advantage == 0:
                    continue

                # 重新计算当前策略下的 log_prob
                gen_ids = responses[i]
                old_log_probs = all_log_probs[i]
                inp = torch.tensor(
                    [tokenizer.encode(prompt).ids], device=device
                )
                curr_log_probs = []
                for token_id in gen_ids:
                    logits = policy(inp)[:, -1, :]
                    probs = F.softmax(logits, dim=-1)
                    curr_log_prob = torch.log(probs[0, token_id])
                    curr_log_probs.append(curr_log_prob)
                    inp = torch.cat(
                        [inp, torch.tensor([[token_id]], device=device)],
                        dim=1
                    )

                # 列表转 tensor
                curr_log_probs = torch.stack(curr_log_probs)
                old_log_probs = torch.stack(old_log_probs).detach()

                # 重要性采样比率
                ratio = torch.exp(curr_log_probs - old_log_probs)

                # PPO 风格的裁剪损失
                surr1 = ratio * advantage
                surr2 = torch.clamp(
                    ratio, 1 - clip_eps, 1 + clip_eps
                ) * advantage
                loss_i = -torch.min(surr1, surr2).sum()
                total_loss += loss_i
                total_tokens += len(gen_ids)
            # ===== 第五步：反向传播，更新策略 =====
            if total_tokens > 0:
                loss = total_loss / total_tokens
                optimizer.zero_grad()
                loss.backward()
                optimizer.step()

        print(f"GRPO epoch {epoch+1} 完成")

    torch.save(policy.state_dict(), "grpo_policy.pth")
    print("GRPO 训练完成！")

if __name__ == "__main__":
    grpo_train()

💡 一、GRPO 是什么？为什么不需要 Critic？

GRPO（Group Relative Policy Optimization） 是 DeepSeek-Math 论文提出的一种 RL 训练方法。它的核心思想可以一句话概括：

🎭 核心创新： 不再训练一个和 Actor 一样大的 Critic 网络来估计"状态价值"，而是对同一个 prompt 采样一组回答，用组内的相对排名来计算优势。

传统 PPO 需要三个网络：Actor（策略）、Critic（价值）、Reward（奖励）。其中 Critic 和 Actor 一样大，训练起来又贵又不稳定。

GRPO 的解法很"江湖"：不需要知道每个人的真实武力值，只需要让同一拨人打一场，赢了加分，输了扣分。

$$ \hat{A}_i = \frac{r_i - \text{mean}(r_1, \dots, r_G)}{\text{std}(r_1, \dots, r_G)} $$

这就是 GRPO 的全部秘密——把 Critic 干掉，用统计学代替神经网络。简单、粗暴、有效。

🗺️ 二、整体架构：组内比武的五个回合

训练过程就像一个擂台赛，每个 prompt 是一场擂台：

步骤操作比喻
1 采样对同一 prompt 采样 G 个回答召集 G 名选手上台
2 打分奖励模型为每个回答打分裁判逐个评分
3 排名组内标准化得到优势值赢了加分，输了扣分
4 更新PPO 裁剪损失反向传播教练根据表现调整训练策略
5 循环遍历所有 prompt，多个 epoch打完一轮再来一轮

⚙️ 三、超参数：比赛规则设定

代码开头定义了一堆超参数，别慌，我们一个个拆：

  • G = 4 —— 每个 prompt 采样 4 个回答。相当于每场擂台赛有 4 名选手。G 越大，排名估计越稳定，但计算量也越大。
  • clip_eps = 0.2 —— PPO 的裁剪阈值。如果策略更新太激进（ratio 偏离 1 超过 20%），就踩刹车。
  • kl_coef = 0.05 —— KL 散度惩罚系数。防止策略跑得太远，保持和 SFT 模型的"血缘关系"。
  • lr = 1e-6 —— 学习率。RL 训练用小学习率是常识，大了会"兴奋过度"。
🤔 为什么 batch_size = 1？ 因为 GRPO 的特殊性——每次处理一个 prompt，然后对这个 prompt 采样 G 个回答。所以真正的 batch 大小是 G × tokens。

📦 四、模型加载：选手入场

这里加载了两个模型，它们的关系很有意思：

  • policy（Actor）：从 SFT 模型初始化，这是我们要训练的主角。
  • ref_model（Reference）：同样从 SFT 初始化，但完全冻结。它是"原版"的参照物，用来计算 KL 散度。
🎭 比喻： policy 就像一个学生，不断学习进步；ref_model 就像他入学时的"档案照"，用来衡量他"变了多远"。如果变化太大（KL 散度太高），就要惩罚。

注意 for param in ref_model.parameters(): param.requires_grad = False，这行确保 Reference 模型永远不会被更新——它是冻结的"锚点"。

1 第一步：采样 G 个回答

这是 GRPO 最关键的一步。对同一个 prompt，用当前策略采样 G 个不同的回答。

代码逻辑：

  • 外层循环 for _ in range(G)：采样 4 次
  • 内层循环 for step in range(50)：每次最多生成 50 个 token
  • 每一步：policy → softmax → 多样性采样 → 记录 log_prob
🎲 关键点： 注意 torch.multinomial(probs, num_samples=1)——这是随机采样，不是 greedy！所以同一个 prompt 采 4 次，会得到 4 个不同的回答。这就是"组"的来源。

同时，每个回答都要记录 log_probs（对数概率），这些是旧策略下的概率，后面算重要性采样比率时要用。

2 第二步：奖励模型打分

采样完成后，把每个回答交给奖励模型打分：

  • 拼接 prompt + response 得到完整输入
  • 奖励模型输出一个标量分数
  • 这个分数可以是正的也可以是负的

注意这里奖励模型是单独的模型，不是 policy 也不是 ref_model。在实际项目中，这个模型通常是人工标注数据训练出来的。

🏋️ 比喻： 如果 policy 是选手，那奖励模型就是裁判。裁判只管打分，不管选手怎么训练——这就是 GRPO 的优雅之处。

3 第三步：组内标准化——代替 Critic 的关键！

这是 GRPO 的灵魂操作，也是它和 PPO 最大的区别。

$$ \hat{A}_i = \frac{r_i - \text{mean}(r)}{\text{std}(r) + \epsilon} $$

三行代码，但含义深远：

  • mean_r = rewards_t.mean()：计算组内平均奖励
  • std_r = rewards_t.std() + 1e-8：计算组内标准差，加 1e-8 防止除零
  • advantages = (rewards_t - mean_r) / std_r：标准化

标准化之后：

  • 得分 高于平均的回答 → 优势为 正 → 鼓励策略多生成这类回答
  • 得分 低于平均的回答 → 优势为 负 → 惩罚策略生成这类回答
  • 恰好等于平均 → 优势为 零 → 不更新
💡 为什么这能代替 Critic？ Critic 的作用是估计"这个状态有多好"，然后算出优势 A = R - V(s)。GRPO 的做法是：不估计绝对值，只比较相对值。同一组内的回答来自同一个 prompt（同一个状态），用组内排名代替绝对价值估计——这在数学上是合理的，因为优势本来就是一个相对量。

这也解释了为什么叫 "Group Relative"——优势是在组内相对计算的。

4 第四步：PPO 风格的裁剪损失

损失计算部分和标准 PPO 几乎一模一样，但有一个关键区别：每个 token 共享同一个句子级优势。

$$ \mathcal{L}^{\text{GRPO}} = -\frac{1}{\sum |y_i|} \sum_{i=1}^{G} \sum_{t=1}^{|y_i|} \min\left( \rho_{i,t} \hat{A}_i,\ \text{clip}(\rho_{i,t}, 1-\epsilon, 1+\epsilon) \hat{A}_i \right) $$

逐步解读代码：

  • 重新计算 log_prob：用当前策略对旧回答重新算概率（因为策略已经更新了）
  • 重要性采样比率：ratio = exp(curr_logp - old_logp)，衡量新旧策略的差异
  • 裁剪：torch.clamp(ratio, 1-ε, 1+ε)，防止更新太猛
  • 取 min：在原始目标和裁剪目标中取较小值——悲观估计，保守更新
  • 除以 token 数：归一化，不同长度的回答不会因为 token 多就主导梯度
🔑 关键区别： 在 PPO 中，优势 A(s_t, a_t) 是逐 token 的（每个 token 有自己的优势估计）。在 GRPO 中，整句话共享同一个 hat{A}_i，因为优势是在句子级别计算的。这意味着好的回答中每个 token 都会被鼓励，差的回答中每个 token 都会被惩罚。

代码中还有一行 if advantage == 0: continue——如果优势恰好为零（等于平均值），就跳过这个回答，不产生梯度。这是一个小优化。

5 第五步：反向传播与训练循环

最后就是标准的 PyTorch 训练流程：

  • loss = total_loss / total_tokens：用总损失除以总 token 数，得到平均损失
  • optimizer.zero_grad()：清空梯度
  • loss.backward()：反向传播
  • optimizer.step()：更新参数

整个训练循环是 epochs × prompts，每个 prompt 产生 G 个回答，最终汇总损失一起更新。

⏱️ 效率对比： 标准 PPO 每步需要 4 个模型前向传播（Actor、Critic、Reward、Reference）。GRPO 只需要 3 个（Policy、Reward、Reference），省掉了和 Actor 一样大的 Critic——内存直接省 25%，训练稳定性还更好。
